## Current Version Note

The main revision outputs were regenerated on 2026-05-18 after aligning the code with the manuscript missing-data description. Binary/count variables are imputed as 0 when missing. Continuous and scale variables are standardized using development-cohort or fold-training mean and SD, and missing standardized values are imputed as 0, equivalent to mean imputation on the original scale. The same development-cohort preprocessing parameters are applied unchanged to the temporal external validation cohort.

Current key results from the regenerated pipeline:

- External validation HybridXGBRF AUROC: 0.898 (95% CI 0.889-0.906).
- External validation HybridXGBRF Brier score: 0.112 (95% CI 0.108-0.116).
- External calibration intercept: 0.516 (95% CI 0.447-0.597).
- External calibration slope: 1.271 (95% CI 1.217-1.342).
- External observed/expected ratio: 1.128 (95% CI 1.108-1.149).
- External Harrell C-index in survival sensitivity analysis: 0.860.

These regenerated values differ from the originally submitted manuscript values. If the revised manuscript uses the regenerated outputs, the Abstract, Results, tables, figure captions, and response letter should be updated consistently.

Facility-size and facility-region outputs are currently most complete for excluded residents because the new `DATA` files retain `dbname` values that can be linked to `DATA/area_size.xlsx`. The Google Sheets analytic-cohort extract used for model development/validation does not provide the same stable resident-level facility linkage after import. Therefore, facility-size and region analyses should be interpreted as facility-level development roster characterization and excluded-resident stratification rather than a complete resident-level included-versus-excluded facility comparison.

The original notebook labels the leading model as `HybridXGBRF (Our Approach)`, but its final `all_models` dictionary appears to assign that label to an `XGBClassifier`. This should be clarified in the manuscript and response letter, especially for SHAP interpretation and claims of superiority over XGBoost.
